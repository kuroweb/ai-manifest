<what-to-do>

このプランについて、共通理解に至るまであらゆる観点から執拗に質問してください。設計ツリーの各分岐を順にたどり、意思決定間の依存を一つずつ解消してください。各質問には推奨回答も示してください。

質問は一度に一つだけ行うこと。各質問へのフィードバックを待ってから次に進むこと。

コードベースを調べれば答えられる質問は、ユーザーに聞く代わりにコードベースを調べること。

## ドメイン認識

コードベースを探索する際は、既存のドキュメントも探すこと。

### ファイル構成

ほとんどのリポジトリは単一のコンテキストを持つ：

```
/
├── CONTEXT.md
├── docs/
│   └── adr/
│       ├── 0001-event-sourced-orders.md
│       └── 0002-postgres-for-write-model.md
└── src/
```

ルートに `CONTEXT-MAP.md` がある場合、リポジトリは複数のコンテキストを持つ。マップは各コンテキストの場所を示す：

```
/
├── CONTEXT-MAP.md
├── docs/
│   └── adr/                          ← システム全体の意思決定
├── src/
│   ├── ordering/
│   │   ├── CONTEXT.md
│   │   └── docs/adr/                 ← コンテキスト固有の意思決定
│   └── billing/
│       ├── CONTEXT.md
│       └── docs/adr/
```

ファイルは遅延作成する — 書く内容があるときだけ作る。`CONTEXT.md` がなければ、最初の用語が解決されたときに作成する。`docs/adr/` がなければ、最初の ADR が必要になったときに作成する。

## セッション中

### 用語集に照らして挑戦する

ユーザーが `CONTEXT.md` の既存の言語と矛盾する用語を使ったら、すぐに指摘する。「用語集では 'cancellation' を X と定義していますが、Y の意味で使っているようです — どちらですか？」

### 曖昧な言葉を研ぎ澄ます

ユーザーが曖昧または多義的な用語を使ったら、正確な正規用語を提案する。「'account' と言っていますが、Customer と User のどちらを指しますか？ これらは別物です。」

### 具体シナリオで議論する

ドメインの関係を議論しているときは、具体的なシナリオでストレステストする。エッジケースを突くシナリオを考え、概念間の境界を明確にさせる。

### コードと突き合わせる

ユーザーが動作の説明をしたら、コードと一致するか確認する。矛盾があれば表面化する。「コードは Order 全体をキャンセルしていますが、部分キャンセルが可能だと言いました — どちらが正しいですか？」

### CONTEXT.md をその場で更新する

用語が解決したら、その場で `CONTEXT.md` を更新する。まとめて書かない — 決まった瞬間に記録する。フォーマットは [CONTEXT-FORMAT.md](./CONTEXT-FORMAT.md) に従う。

`CONTEXT.md` には実装の詳細を一切書かない。仕様書、メモ帳、実装判断の置き場として扱わない。用語集であり、それだけである。

### ADR は控えめに提案する

次の3つがすべて当てはまるときだけ、ADR の作成を提案する：

1. **取り消しが難しい** — 後から方針を変えるコストが意味のあるもの
2. **文脈なしでは驚く** — 将来の読者が「なぜこうしたのか？」と疑問に思う
3. **実際のトレードオフの結果** — 本当の代替案があり、特定の理由で一つを選んだ

3つのうち1つでも欠けていれば、ADR はスキップする。フォーマットは [ADR-FORMAT.md](./ADR-FORMAT.md) に従う。
