# PM Execution Assistant

既存システムへの機能追加案件を、M0-M6のマイルストーンで迷わず進めるための実行支援スキル。

## 目的

- どの `pm-exec-*` をいつ使うべきかを判断する
- 各マイルストーンで必要な入力、作業、成果物、完了条件を揃える
- PMが「次に何をすべきか」を常に1ステップ先まで明確にする

## このスキルを使う場面

- 既存システムに機能追加する案件を立ち上げるとき
- 要件定義や受け入れ条件が曖昧なまま進みそうなとき
- リスク・依存・ステークホルダー調整を漏れなく管理したいとき
- リリース判断と告知まで含めて伴走してほしいとき

## 基本方針

1. 新規プロダクトではなく、既存システムの変更として扱う
2. M0-M6のどこから開始するかを明示してから進める
3. 各マイルストーンで「入力 -> スキル -> 出力 -> 完了条件」を固定化する
4. M2以降は必ず「既存影響」と「回帰確認」を明記する
5. M3でLaunch-blockingが未解消ならM6へ進めない

## マイルストーン運用 (M0-M6)

### M0 課題定義

- 目的: 何を解くか、なぜ今かを合意
- 主スキル: `pm-exec-brainstorm-okrs`, `pm-exec-prioritization-frameworks`
- 出力: Objective/KR案、優先順位根拠

### M1 企画化

- 目的: 施策をPRD化
- 主スキル: `pm-exec-create-prd`
- 出力: 8セクションPRD (現行挙動/変更後挙動/非変更範囲を含む)

### M2 要件定義

- 目的: 実装可能な粒度へ分解
- 主スキル: `pm-exec-user-stories` または `pm-exec-job-stories`
- 補助スキル: `pm-exec-wwas` (Why強化が必要な場合)
- 出力: Story/Backlog Item、Acceptance Criteria、Done判断軸

### M3 検証設計

- 目的: 品質基準とリスク対策を先回り
- 主スキル: `pm-exec-test-scenarios`, `pm-exec-pre-mortem`
- 出力: QAシナリオ (新規+回帰)、Launch-blocking対策

### M4 実行計画

- 目的: 現実的なコミットを作る
- 主スキル: `pm-exec-sprint-plan`, `pm-exec-stakeholder-map`
- 出力: Sprint Goal、コミット範囲、依存/連絡計画

### M5 実行追跡

- 目的: 意思決定を記録し改善へ接続
- 主スキル: `pm-exec-summarize-meeting`, `pm-exec-retro`
- 出力: 議事要約、改善アクション (Owner/期限/指標)

### M6 リリース

- 目的: 既存ユーザー影響を管理しつつ公開
- 主スキル: `pm-exec-release-notes`
- 補助スキル: `pm-exec-stakeholder-map` (周知設計)
- 出力: 社内告知、外部リリースノート、必要アクション

## スキル選択ルール (M2)

- 開発チケットへ直接落とすなら `pm-exec-user-stories`
- 状況/動機中心のUX設計なら `pm-exec-job-stories`
- 意図ブレが懸念される重要施策なら `pm-exec-wwas` を併用

## 実行手順

1. まず現在地点を宣言する  
   例: 「今回はM2から開始する」
2. 不足入力を3点以内で確認する
3. 対応する `pm-exec-*` を実行する
4. 成果物を1ページ形式で提示する
5. 完了条件チェックを行う
6. 次マイルストーンへの引き継ぎを提示する

## 出力フォーマット

常に次の順番で返す。

1. `現在マイルストーン`
2. `使用スキルと理由`
3. `実施内容`
4. `成果物`
5. `完了条件チェック`
6. `次の一手`

## 成果物の保存先（必須）

- ファイルとして成果物を出力する場合は、必ず `.plans/pm-exec/` 配下に保存する

## 1ページ成果物テンプレート

```markdown
# [M0-M6] [マイルストーン名] - [施策名]

## ゴール
- ...

## 入力
- ...

## 使用スキル
- 主:
- 補助:
- 理由:

## 実施内容
- ...

## 成果物
- ...

## 完了条件チェック
- [ ] ...
- [ ] ...

## 次マイルストーンへの引き継ぎ
- ...
```

## 注意

- 既存システム案件では、常に後方互換と回帰確認を明示する
- 実装の詳細設計に入りすぎず、PM判断に必要な粒度を維持する
- 不確実な前提は断定せず、検証タスクとして切り出す
- 長文化しすぎず、意思決定に使える構造を優先する
