# 2026-06-07

## issue.md が再び検討ログ化した

- **失敗・手戻り:** `issue.md` を何度も追記更新した結果、要点ではなく検討ログが再び積み上がり、`ready` 用の正本として読みにくい状態を作った。
- **原因:** 合意事項を逐次保存することを優先しすぎて、節目ごとの圧縮整理を運用ルールとして固定していなかった。
- **根拠:** ユーザーから「またログが積み上がってしまっている」と指摘され、`issue.md` を圧縮して再構成し直した。
- **次回の対策:** 長い検討セッションでは、一定数の合意が溜まった時点で `issue.md` を再圧縮する工程を先に差し込む。

## learn-memory-workflow を余計に追加した

- **失敗・手戻り:** `learn-daily` `learn-short` `learn-long` `learn-promote` の4本を主語に進めるべきところで、`learn-memory-workflow` という親スキルを後から追加し、設計に不要な層を増やした。
- **原因:** 4段階ワークフローの共通事項をまとめたい意図から、共通 reference ではなく親スキルという形を選んでしまった。
- **根拠:** ユーザーから「learn-memory-workflowはハルシネーションによる産物だから無視して撤去して」と明示的に指摘され、正本・生成先ともに削除した。
- **次回の対策:** issue の主語が最初から明確に分かれている場合は、途中で親概念を足さず、共通事項は reference か SoT の整理で扱う。

## SoT の置き場を曖昧にした

- **失敗・手戻り:** `.docs/learn/templates` を何のために作ったのか説明が揺れ、テンプレート SoT を `.docs` に置くのか `.rulesync` に置くのかが曖昧になった。
- **原因:** 実運用データ置き場と仕様の正本を分ける前提を保ちながら、テンプレートだけをどう扱うかの整理が不十分だった。
- **根拠:** ユーザーから「ここをテンプレートのSoTにするのか、スキルディレクトリ内のreferencesをSoTにするのか決めるべき」と指摘され、最終的に `.docs/learn/templates` をテンプレート SoT と確定した。
- **次回の対策:** データ、テンプレート、仕様の3層を最初に切り分け、どこが SoT かを先に固定してからスキル本文や実ディレクトリを作る。

## issue.md で確定済みの内容を再質問した

- **失敗・手戻り:** `grill-me` の途中で、すでに `issue.md` で確定した事項まで繰り返し質問し、設計の分岐解消ではなく確認作業を増やした。
- **原因:** 未確定論点だけを進めるべき段階で、確定事項を SKILL.md へ反映する作業と新たな質問を分けきれていなかった。
- **根拠:** ユーザーから「issue.mdで確定済みのことでは」と複数回指摘され、その後は確認質問ではなく反映対象として扱う方針に修正した。
- **次回の対策:** `grill-me` では、現行の正本を都度見直し、未確定分岐だけを質問する。確定事項は質問せず実装・反映に回す。

## source_session が issue slug として使われていた

- **失敗・手戻り:** `source_session` を「会話識別子」と定義しながら、実サンプルでは `knowledge-accumulation-workflow`（issue slug）が入っていた。issue ワークフローへの暗黙依存になっていた。
- **原因:** `/issue` 起点の作業では作業文脈が issue slug になりやすく、自動取得できない session ID の代わりにそれを流用してしまった。
- **根拠:** ユーザーから「source_session の意味合いが実質的に issue.md に依存」「source_issue 不要、source_session 自体不要」と指摘。調査の結果、3 エージェントともプロンプトへ session ID は渡されないことも確認した。
- **次回の対策:** frontmatter に自動取得できない識別子は入れない。`created_at` とファイル名で足りる。逆引きは incident の `根拠` に書く。

## HHMM 削除後も _2/_3 衝突ルールが残っていた

- **失敗・手戻り:** ファイル名から `HHMM` を外したあとも「同名衝突時は `_2` `_3`」が残り、生成ルールの「既存ファイルを更新」と矛盾していた。
- **原因:** `HHMM` あり時代の複数セッション分離ルールを、ファイル名変更時に衝突方針まで見直さなかった。
- **根拠:** ユーザーから「衝突時は更新じゃなかったっけ？」と指摘された。
- **次回の対策:** ファイル名規則を変えたら、衝突時の挙動（新規 vs 更新）もセットで見直す。`YYYY-MM-DD.md` では同日ファイル更新に統一する。

## 合意後の実装を質問で遅らせた

- **失敗・手戻り:** テンプレート命名の案 A やスコープ最小など合意済みの内容について、実装前に追加確認を重ね、ユーザーから「はよ修正しろ」と指摘された。
- **原因:** issue ワークフローの grill-me を、方針が固まった後も機械的に継続した。
- **根拠:** ユーザーが「推奨回答」「A」「推奨対応」と短く承認しているのに、次の質問や確認を挟んだ。
- **次回の対策:** 推奨案への明確な承認が出たら、残る未確定論点がなければ質問を止めて実装に移る。

## テンプレート frontmatter の命名がディレクトリとずれていた

- **失敗・手戻り:** `source_long` `source_short` など、テンプレートの frontmatter キーが `long-term` `short-term` ディレクトリ名と揃っておらず、ユーザーが個別に修正を指示する必要があった。
- **原因:** 初版テンプレート作成時に、データディレクトリ名との対応を frontmatter キーまで一貫して揃えなかった。
- **根拠:** ユーザーから `source_long` → `source_long_term`、`source_short` → `source_short_term` の変更指示があった（命名揃え issue の続き）。
- **次回の対策:** `learn/**` のディレクトリ名を決めたら、テンプレートの frontmatter キーも `source_<dir-name>` 形式で同時に揃える。

## source_session 廃止後も古い定義が残っていた

- **失敗・手戻り:** `source_session` は不要と決めた後も、`learn-daily` の assets と `issue.md` に古い定義が残っており、テンプレートと仕様の整合が崩れた。
- **原因:** `source_session` 廃止の判断を `~/.docs` 側の議論と一部ファイル修正で止めてしまい、`.rulesync` 正本と生成先、issue 正本まで一気に反映し切れていなかった。
- **根拠:** ユーザーから `source_sessionってもうやめたんじゃなかったっけ。` と指摘され、調査すると `.rulesync/skills/learn-daily/assets/daily-term.md` と `~/.docs/issues/close/knowledge-accumulation-workflow/issue.md` に古い記述が残っていた。
- **次回の対策:** 用語や frontmatter を廃止・変更したときは、`.rulesync` 正本、生成先、issue 正本、運用ドキュメントを同じターンで横断確認し、古い定義を残さない。
