# 2026-06-07_0009_codex

## issue.md が再び検討ログ化した

- **失敗・手戻り:** `issue.md` を何度も追記更新した結果、要点ではなく検討ログが再び積み上がり、`ready` 用の正本として読みにくい状態を作った。
- **原因:** 合意事項を逐次保存することを優先しすぎて、節目ごとの圧縮整理を運用ルールとして固定していなかった。
- **根拠:** ユーザーから「またログが積み上がってしまっている」と指摘され、`issue.md` を圧縮して再構成し直した。
- **次回の対策:** 長い検討セッションでは、一定数の合意が溜まった時点で `issue.md` を再圧縮する工程を先に差し込む。

## learn-memory-workflow を余計に追加した

- **失敗・手戻り:** `learn-daily` `learn-short` `learn-long` `learn-promote` の4本を主語に進めるべきところで、`learn-memory-workflow` という親スキルを後から追加し、設計に不要な層を増やした。
- **原因:** 4段階ワークフローの共通事項をまとめたい意図から、共通 reference ではなく親スキルという形を選んでしまった。
- **根拠:** ユーザーから「learn-memory-workflowはハルシネーションによる産物だから無視して撤去して」と明示的に指摘され、正本・生成先ともに削除した。
- **次回の対策:** issue の主語が最初から明確に分かれている場合は、途中で親概念を足さず、共通事項は reference か SoT の整理で扱う。

## SoT の置き場を曖昧にした

- **失敗・手戻り:** `.docs/learn/templates` を何のために作ったのか説明が揺れ、テンプレート SoT を `.docs` に置くのか `.rulesync` に置くのかが曖昧になった。
- **原因:** 実運用データ置き場と仕様の正本を分ける前提を保ちながら、テンプレートだけをどう扱うかの整理が不十分だった。
- **根拠:** ユーザーから「ここをテンプレートのSoTにするのか、スキルディレクトリ内のreferencesをSoTにするのか決めるべき」と指摘され、最終的に `.docs/learn/templates` をテンプレート SoT と確定した。
- **次回の対策:** データ、テンプレート、仕様の3層を最初に切り分け、どこが SoT かを先に固定してからスキル本文や実ディレクトリを作る。

## issue.md で確定済みの内容を再質問した

- **失敗・手戻り:** `grill-me` の途中で、すでに `issue.md` で確定した事項まで繰り返し質問し、設計の分岐解消ではなく確認作業を増やした。
- **原因:** 未確定論点だけを進めるべき段階で、確定事項を SKILL.md へ反映する作業と新たな質問を分けきれていなかった。
- **根拠:** ユーザーから「issue.mdで確定済みのことでは」と複数回指摘され、その後は確認質問ではなく反映対象として扱う方針に修正した。
- **次回の対策:** `grill-me` では、現行の正本を都度見直し、未確定分岐だけを質問する。確定事項は質問せず実装・反映に回す。
