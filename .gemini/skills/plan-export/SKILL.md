# プラン出力

エージェントと相談して固めたプラン内容を、後で参照しやすい Markdown ファイルとして `~/.plans` に保存する。

このスキルの優先事項は、会話の中でできたプラン本文を失わず、そのまま再利用できる形で保存すること。整形は必須ではない。

## トリガー

- `/plan-export`
- 「プランを保存して」
- 「相談したプランを `~/.plans` に出力して」
- 「この計画を書き出して」

## 手順

1. 現在の会話から、合意済みのプラン内容だけを抽出する
2. プラン本文がすでに Markdown として十分に読める場合は、原文をそのまま使う
3. 不完全なメモしかない場合だけ、最小限の見出しや箇条書きで整理する
4. 未確定事項と決定事項を分け、曖昧な点は推測で埋めない
5. `~/.plans/` ディレクトリがなければ作成する
6. `YYYY-MM-DD_HHmm_<slug>.md` の形式で保存する
7. 同名ファイルがある場合は末尾に `_2` などの連番を付ける

`<slug>` はプランの主題を 3 語前後で短く表した ASCII のケバブケースにする。適切な主題が決めにくい場合は `plan` を使う。

## 保存時の注意

- `~/.plans` への書き込みは環境によってサンドボックス外になることがある
- その場合は権限昇格が必要な旨を明示し、承認付きで保存する
- 権限がなく保存できない場合でも、保存予定の本文はそのまま生成して失わない

## 出力方針

### 原則

- 既存のプラン本文が完成しているなら、コピペ保存を優先する
- 整形は読みやすさを明確に改善する場合にだけ行う
- 会話に出ていない内容は補わない
- 原文のニュアンスや優先順位を変えない

### そのまま保存してよいケース

- `plan` モードや planner エージェントの出力が、すでに見出し付き Markdown になっている
- ユーザーが「この内容をそのまま保存して」と明示している
- 追記なしで次回参照できるだけの情報量がある

### 最小限の整形をしてよいケース

- 会話断片しかなく、保存しても再利用しにくい
- 箇条書きの順序や区切りを付けるだけで可読性が大きく上がる
- 未確定事項を別立てにしないと誤読しやすい

## 整形が必要な場合の最小テンプレート

### `# 概要`

- プラン全体を 3〜6 個の箇条書きで要約する

### `# 背景`

- なぜこのプランが必要か
- 何を解決するための計画か

### `# スコープ`

- 対象
- 対象外

### `# 実施ステップ`

- 実行順に並べる
- 可能なら各ステップの完了条件を一行で添える

### `# 懸念点・リスク`

- 想定される失敗要因
- 判断待ち事項

### `# 未確定事項`

- ユーザー確認が必要な点
- まだ結論を出していない点

### `# 次のアクション`

- 直近でやることを優先順で書く

## ルール

- 実装詳細よりも、意思決定と段取りが再開できることを優先する
- 冗長な議事録にはせず、次回すぐ着手できる粒度で整理する
- ユーザーや別エージェントがそのまま読める、簡潔な Markdown にする
