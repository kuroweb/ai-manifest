# Code Investigator

既存コードを調査し、原因・影響範囲・対応方針を明らかにする調査スキル。
調査の手順やツール選定は状況に応じて最適化してよいが、最終出力はこのファイルで定義する要求を必ず満たすこと。

## 成果物としての必須要件

- 意思決定可能な調査レポートであること（読者が次アクションを判断できる）
- 主要主張ごとに、対応するコード根拠があること
- コード根拠はリポジトリ相対パスとシンボル名で示すこと
- 影響範囲を直接影響と間接影響に分けること
- 未調査領域を明示すること
- 対応方針を優先順位付きで示すこと

## 実行開始時に最初に確定する項目

1. 調査目的（何を判断するための調査か）
2. 調査範囲（対象機能、対象モジュール、除外範囲）
3. 調査読者（実装者、レビュアー、意思決定者）
4. 完了条件（何が分かれば調査完了か）

不足情報がある場合は、仮置きせず確認質問を返す。

## レポート要件

このセクションの要件はすべて必須。

### 1) 要約

- 先頭に「結論サマリ」を置く
- 結論、根拠、推奨アクションを3点以内で示す
- 読者が1分で状況を把握できる密度にする

### 2) 根拠提示

- 主要主張に対して、最低1つ以上のコード根拠を付ける
- コード根拠は「相対パス + シンボル名」で示す
- 可能な限り行番号も併記する
- 1主張1根拠以上の対応関係を崩さない

### 3) 影響範囲

- 直接影響（壊れる・変わる箇所）
- 間接影響（依存先、周辺フロー、運用）
- 未調査領域（今回見ていない範囲）

### 4) 対応方針

- 最優先で実施すること
- 次点で実施すること
- 見送り判断と理由
- 高リスク項目は `⚠️` を付ける

## レポートテンプレート

以下の構造を維持すること。

```markdown
# 目的
- 背景:
- 判断したいこと:
- 調査範囲:
- 非対象:

# 結論サマリ
- 結論:
- 根拠1:
- 根拠2:
- 根拠3:
- 👉推奨アクション:

# 調査詳細
## 論点1
- 観測事実:
- コード根拠: `src/...`（相対パス, 関数/クラス名）
- 補助根拠（ログ/クエリ/設定）:
- 解釈:

## 論点2
- 観測事実:
- コード根拠: `src/...`（相対パス, 関数/クラス名）
- 補助根拠（ログ/クエリ/設定）:
- 解釈:

# 影響範囲
- 直接影響:
- 間接影響:
- 未調査:
- ⚠️注意点:

# 対応方針
- 最優先:
- 次点:
- 見送りと理由:
```

## 出力品質チェック

- [ ] 要約だけで意思決定の前提が分かる
- [ ] 主要主張にコード根拠がある
- [ ] コード根拠が「相対パス + シンボル名」形式になっている
- [ ] 影響範囲が直接/間接で分離されている
- [ ] 未調査領域が明記されている
- [ ] 推測と事実が混ざっていない
- [ ] 次アクションが担当者目線で実行可能
