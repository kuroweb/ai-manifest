# Claude Code 権限ルール・棚卸し（提案のみ）

デバッグログや貼り付けテキストから、**観測された事実の一覧**と **`permissions` への候補ルール案**を出す。  
**`settings.json` や `settings.local.json` を勝手に書き換えない。** 採用はユーザーが判断する。

## ゴール

1. **リスト棚卸し**: どのツール・コマンド・パス・ドメイン・MCP が現れたかを表形式で整理する。
2. **提案**: `allow` / `deny` / `ask` に載せうる文字列（`Tool` または `Tool(specifier)`）を候補として列挙する。
3. **リスク注記**: 広すぎるパターン（例: 安易な `Bash` 丸ごと）や、Read/Edit ルールと Bash の穴を明示する。

## 入力の取り方

次から適宜組み合わせる。無いものは「なし」で進める。

| 入力 | 例 |
| --- | --- |
| デバッグログ | `~/.claude/debug/` 配下のセッションログ、`latest` の指すファイル、`CLAUDE_CODE_DEBUG_LOGS_DIR` 先 |
| 貼り付け | ユーザーが抜粋したログ（長い場合は関連行のみでも可） |
| 現行設定 | `.claude/settings.json` / `settings.local.json` の `permissions` ブロック（読める場合） |
| 公式 UI | Claude Code の `/permissions` で確認できるルールのメモ（あれば） |

ログ行の形式は **Claude Code のバージョンで変わりうる**。最初に数十行を読み、どのキーワード・JSON 断片がツール呼び出しに対応するかを仮定し、矛盾があれば仮定を修正する。

## 手順

1. **スコープ確定**: 対象ログ・期間・プロジェクト（単一リポジトリか）をはっきりさせる。ログが無い場合はユーザーへの聞き取りから始め、推測ルールは後述のルールに従い「ログ外の一般的推奨」として分離する。
2. **ファクト抽出**: 次のカテゴリで **重複を除いた一覧** と **出現回数または代表例** を作る（ログから読み取れない列は「不明」）。
   - **Bash**: コマンド先頭（引数まで取れるならその範囲）
   - **Read / Edit / Write / Grep / Glob** 等: パスやパターン
   - **WebFetch**: URL またはドメイン
   - **MCP**: `mcp__サーバ名__ツール名` 形式の候補
   - **Agent**: サブエージェント名がログにあれば
3. **既存ルールとの突合**: 現行の `allow` / `deny` / `ask` があれば、観測と照らして「既にカバー」「不足」「過剰」のメモを付ける。
4. **候補ルールの提案**: 各行に以下を付ける。
   - 提案ルール文字列（[公式の Permission rule syntax](https://code.claude.com/docs/en/permissions) に沿う）
   - **推奨配置**: `allow` / `deny` / `ask` のどれか（複数案がある場合は優先順）
   - **根拠**: ログのどの観測に基づくか（一行でよい）
   - **リスク**: ワイルドカードの広がり、Bash 経由で Read 制限をすり抜ける点など
5. **やらないことの確認**: 自動修正・ファイル保存は行わない。採用時はユーザーが手でマージする。

## 出力フォーマット（この順で出す）

**B と C の役割**: **B は読みやすい候補リスト（箇条書き・根拠付き）だけ。** ユーザーが `settings.json` にそのまま貼る **コメントなし JSON は C を1つだけ**出す（C の配列には、B で採用する行だけ入れる）。

### A. 観測サマリ

表（カテゴリ、件数、代表例、備考）。

### B. `permissions` 候補リスト（説明用・ここでは JSON にしない）

- **allow 候補**（各行: ルール文字列、根拠、リスクの一行メモ）
- **deny 候補**（同様）
- **ask 候補**（毎回確認したいもの）

### C. 参考 JSON 片（提案・未適用・コピー用はここだけ）

ユーザーが手で貼る用に、次の形の **コメントなし JSON を1つだけ**示す。編集はユーザー任務。

```json
{
  "permissions": {
    "allow": [],
    "deny": [],
    "ask": []
  }
}
```

配列には **B でユーザーが採用すると決めた候補だけ**を入れる想定と明記する。`ask` を使わないならキーごと省略してよいことも書く。

### D. 確認チェックリスト

- 広すぎる `Bash(*)` や安易な `Bash` だけの allow になっていないか
- ネットワーク制限を `WebFetch` だけに頼っていて、許可した `Bash` から `curl` 等が漏れていないか
- `Read(./.env)` の deny があっても **Bash で `cat .env` は防げない**ことの認識があるか
- 評価順 **deny → ask → allow** で意図どおりか

## 短い入出力例（架空・実ログの形式とは一致しない場合がある）

### 入力（ログ抜粋の例）

```
… tool Bash … npm run test …
… tool Read … package.json …
… tool Bash … git push origin main …
```

### 出力イメージ（構造の見本）

**A. 観測サマリ（表の例）**

| カテゴリ | 件数 | 代表例 | 備考 |
| --- | --- | --- | --- |
| Bash | 2 | `npm run test`, `git push origin main` | |
| Read | 1 | `package.json` | |

**B. 候補リスト（例）**

- allow 候補: `Bash(npm run *)` — テストが繰り返し実行された（ワイルドカードは `npm run` 以外も含むので要判断）
- deny 候補: `Bash(git push *)` — リモートへの push は都度確認したい
- ask 候補: `Edit` — 本番直前は毎回確認したい場合

**C. 参考 JSON（B のうち採用したものだけ配列へ）**

```json
{
  "permissions": {
    "allow": ["Bash(npm run *)"],
    "deny": ["Bash(git push *)"],
    "ask": ["Edit"]
  }
}
```

## ルール（スキル利用時の拘束）

- **設定ファイルを自動では書き換えない**（ユーザーが「この内容で書き込んで」と明示した場合のみ、その指示に従う）。
- ログに無いことは **推測で候補ルールにしない**。必要なら「ログ外の一般的推奨」として別小節に分け、推測であると書く。
- 公式ドキュメントと矛盾する書き方を提案しない。不明な点は不明と書く。
