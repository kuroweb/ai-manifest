# ADR フォーマット

ADR は `docs/adr/` に置き、連番で命名する：`0001-slug.md`、`0002-slug.md` など。

`docs/adr/` ディレクトリは遅延作成する — 最初の ADR が必要になったときだけ作る。

## テンプレート

```md
# {意思決定の短いタイトル}

{1〜3文：文脈は何か、何を決めたか、なぜそうしたか。}
```

以上。ADR は1段落で十分。価値は、意思決定が行われたことと、その理由を記録することにある — セクションを埋めることではない。

## 任意セクション

本当に価値があるときだけ含める。ほとんどの ADR には不要。

- **Status** フロントマター（`proposed | accepted | deprecated | superseded by ADR-NNNN`）— 意思決定を見直すときに有用
- **Considered Options** — 却下した代替案を覚えておく価値があるときだけ
- **Consequences** — 自明でない下流の影響を明示する必要があるときだけ

## 採番

`docs/adr/` の既存ファイルで最大の番号を探し、1つ増やす。

## ADR を提案するタイミング

次の3つがすべて当てはまること：

1. **取り消しが難しい** — 後から方針を変えるコストが意味のあるもの
2. **文脈なしでは驚く** — 将来の読者がコードを見て「なぜこんなことを？」と疑問に思う
3. **実際のトレードオフの結果** — 本当の代替案があり、特定の理由で一つを選んだ

取り消しが容易ならスキップする — そのときは取り消せばよい。驚かないなら、誰も理由を疑問に思わない。本当の代替がなければ、「当たり前のことをした」以上に記録するものはない。

### 該当する例

- **アーキテクチャの形。** 「モノレポを使う。」「書き込みモデルはイベントソーシング、読み取りモデルは Postgres に投影する。」
- **コンテキスト間の統合パターン。** 「Ordering と Billing は同期的 HTTP ではなくドメインイベントで通信する。」
- **ロックインを伴う技術選択。** データベース、メッセージバス、認証プロバイダ、デプロイ先。すべてのライブラリではなく、入れ替えに四半期かかるものだけ。
- **境界とスコープの意思決定。** 「顧客データは Customer コンテキストが所有し、他コンテキストは ID のみ参照する。」明示的な no は yes と同じくらい価値がある。
- **自明な道からの意図的な逸脱。** 「X の理由で ORM ではなく手書き SQL を使う。」合理的な読者が逆を想定するもの。次のエンジニアが意図的な選択を「修正」してしまうのを防ぐ。
- **コードから見えない制約。** 「コンプライアンス要件で AWS は使えない。」「パートナー API 契約のため応答時間は 200ms 未満である必要がある。」
- **却下理由が自明でない代替案。** GraphQL を検討して微妙な理由で REST を選んだなら記録する — さもないと半年後にまた GraphQL が提案される。
